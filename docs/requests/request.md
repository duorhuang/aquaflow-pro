## 3. 双端同步机制 (Sync Engine)

| # | 同步场景 | 期望行为 | 状态 |
|---|----------|----------|------|
| 3.1 | 教练发布训练 -> 队员端可见 | 教练点击"发布"后，队员端在 30s 内自动拉取到新计划 | ⚠️ 待验证（代码已部署） |
| 3.2 | 队员提交反馈 -> 教练端可见 | 队员提交周总结/日反馈后，教练端在 30s 内自动看到 | ⚠️ 待验证（代码已部署） |
| 3.3 | 教练打卡 -> 队员端日历更新 | 教练标记出勤后，队员端出勤日历自动反映 | ⚠️ 待验证（代码已部署） |
| 3.4 | 队员更新状态 -> 教练端监控面板更新 | 队员修改 Readiness/状态后，教练端 `SwimmerStatusPanel` 自动更新 | ⚠️ 待验证（代码已部署） |
| 3.5 | 教练回复评语 -> 队员端可见 | 教练写完评语后，队员在"教练的批复"面板中看到 | ⚠️ 待验证（代码已部署） |
| 3.6 | 教练发起专项问卷 -> 队员端收到通知 | 教练发送问题后，队员端"反馈"Tab 出现待办红点 | ⚠️ 待验证（代码已部署） |
| 3.7 | 变动保护锁（Mutation Guard） | 本地修改后 15 秒内不被服务端旧数据覆盖 | ⚠️ 待验证（代码已部署） |

> [!CAUTION]
> **同步功能是当前系统最大的阻塞问题。** 以上所有同步场景代码层（30s轮询+防御锁）**已经全部写完**。
> 目前唯一的卡点是由于自定义域名 `sw.sportsflow.best` 未绑定到最新的 Worker 版本，导致真实流量未能打到新代码上。
> 等待修复了 Custom Domain 路由后，即可进行联调测试。
> 本地代码中的同步逻辑（30s 轮询 + Mutation Guard）在代码层面已实现，但必须在成功部署后进行端到端实测验证。

### 同步技术实现
- **机制**：30 秒轮询（`setInterval`），非 WebSocket
- **防冲突**：15 秒 Mutation Guard（`recordMutation()`）
- **乐观更新**：所有写操作先更新本地 UI、再异步发送至服务器
- **回滚**：网络失败时回滚本地状态（出勤、队员添加等）
