## Critical Findings

### C-1: `planRepo.update()` double JSON-encodes `blocks` and `targetedNotes` — data corrupts after first edit
**File:** `lib/repos/plans.ts:34-35`  
**Severity:** CRITICAL — Data corruption  
**Layer:** Architecture  
**Impact:** Every plan that a coach edits MORE than once will have corrupted `blocks` and `targetedNotes`. The athlete's training view will show broken/empty blocks.

**Root Cause:**
```ts
// Line 34-35 — update() method
const blocksJson = data.blocks !== undefined ? this.toJson(data.blocks) : (current as any).blocks;
const targetedNotesJson = data.targetedNotes !== undefined ? this.toJson(data.targetedNotes) : (current as any).targetedNotes;
```
`parseJsonRows()` (line 14 in `list()`) parses JSON strings back to objects. But `update()` fetches with `SELECT *` (raw), where `blocks` is already a JSON string like `"[{\"id\":\"x\"}]"`. When this raw string is used as a SQL parameter, Neon calls `JSON.stringify()` on it, producing `"\"[{\\\"id\\\":\\\"x\\\"}]\""` — double-encoded.

The `create()` path is safe (always calls `this.toJson()`). The bug ONLY manifests on the second+ edit.

**Patch (`lib/repos/plans.ts` lines 34-35):**
```ts
const blocksJson = data.blocks !== undefined
  ? this.toJson(data.blocks)
  : (typeof (current as any).blocks === 'string'
    ? (current as any).blocks
    : this.toJson((current as any).blocks));
const targetedNotesJson = data.targetedNotes !== undefined
  ? this.toJson(data.targetedNotes)
  : (typeof (current as any).targetedNotes === 'string'
    ? (current as any).targetedNotes
    : this.toJson((current as any).targetedNotes));
```

### C-2: `/settings` page has NO middleware auth guard — athletes can access coach settings
**File:** `middleware.ts:29-43`  
**Severity:** CRITICAL — Authorization bypass  
**Layer:** Architecture  
**Impact:** Any authenticated user (athlete or coach) can navigate directly to `/settings` and access coach-only data management functions (export, import, clear data, reset demo data).

**Root Cause:**
The middleware checks:
1. Public routes → `/settings` is NOT public
2. Coach routes (`pathname.startsWith('/dashboard')`) → `/settings` does NOT start with `/dashboard`
3. Athlete routes (`/workout`, `/athlete`, `/profile`, `/archive`) → `/settings` not matched
4. Falls through to `NextResponse.next()` → **NO PROTECTION**

The settings page is at `app/(driver)/settings/page.tsx` which resolves to `/settings`. The `(driver)` route group does not affect the URL path.

**Patch (`middleware.ts` — after line 43):**
```ts
// Also protect /settings as a coach-only route
if (pathname === '/settings' || pathname.startsWith('/settings')) {
  if (!token) {
    const loginUrl = new URL('/login', request.url);
    loginUrl.searchParams.set('role', 'coach');
    loginUrl.searchParams.set('redirect', pathname);
    return NextResponse.redirect(loginUrl);
  }
  const payload = await verifyJWT(token);
  if (!payload || payload.role !== 'coach') {
    return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
  }
  return NextResponse.next();
}
```

### C-3: Sync engine polling NEVER recovers from DB offline state
**File:** `lib/store/sync-engine.ts:104-133`  
**Severity:** CRITICAL — App stuck offline  
**Layer:** Architecture  
**Impact:** If the database quota is exhausted during initial load or any polling cycle, `offlineRef.current` is set to `true` and polling stops forever. The app shows stale data and never recovers even when the DB comes back online — user must refresh the page.

**Root Cause:**
```ts
// Line 105 — polling interval
if (offlineRef.current) return;  // Always returns, never retries
```

The recovery check at lines 84-88 is inside `loadData()` which only runs ONCE on mount. The polling interval at line 104 returns immediately when `offlineRef.current` is true, so it never attempts to re-fetch.

**Patch (`lib/store/sync-engine.ts` — modify polling at line 104):**
```ts
const syncInterval = setInterval(async () => {
  // Skip during mutation guard window
  if (Date.now() - lastMutationAt.current < MUTATION_GUARD_MS) return;

  setSyncStatus('syncing');
  try {
    let syncData: any = null;
    try {
      syncData = await fetchAPI('/sync', undefined, true, 1);
    } catch (e: any) {
      if (isQuotaError(e.message)) {
        if (!offlineRef.current) {
          console.warn('[DB] Quota exceeded — falling back to local data');
          offlineRef.current = true;
          setDbOffline(true);
        }
        setSyncStatus('idle');
        return;
      }
    }

    // DB back online — recovery path
    if (syncData && offlineRef.current) {
      console.log('[DB] DB back online — resuming normal sync');
      setDbOffline(false);
      offlineRef.current = false;
    }

    if (syncData) onSync(syncData);
    setSyncStatus('idle');
  } catch {
    setSyncStatus('error');
  }
}, POLL_INTERVAL_MS);
```

### H-3: `isQuotaError` too broad — "exceeded" substring matches false positives
**File:** `lib/store/sync-engine.ts:14-22`  
**Severity:** HIGH  
**Impact:** Any error containing "exceeded" (rate limit exceeded, token limit exceeded, etc.) incorrectly sets `dbOffline = true`, stopping all polling.

**Patch:**
```ts
export function isQuotaError(msg: string): boolean {
  return (
    msg?.includes('data transfer quota') ||
    msg?.includes('HTTP status 402') ||
    msg?.includes('QUOTA-EXHAUSTED') ||
    msg?.includes('Database Quota Exceeded')
  );
}
```
Remove `"exceeded"` (too broad) and `"API Error: 503"` (any 503 is not quota error).

### H-5: `deleteAnnouncement` rollback is a no-op — deleted item not restored
**File:** `lib/store/entity-crud.ts:295-296`  
**Severity:** HIGH  
**Impact:** If the API delete fails, the announcement stays removed from local state. The user sees it deleted even though it still exists on the server.

**Patch:**
```ts
const deleteAnnouncement = useCallback(async (id: string) => {
  const ann = announcements.find(a => a.id === id);  // <-- Save reference
  recordMutation();
  setAnnouncements(prev => { const next = prev.filter(a => a.id !== id); persist('announcements', next); return next; });
  try { await api.announcements.delete(id); } catch {
    if (ann) setAnnouncements(prev => { const next = [...prev, ann]; persist('announcements', next); return next; });
  }
}, [announcements, recordMutation]);
```

### H-6: Athlete workout page uses `toISOString()` for date comparison — timezone off-by-one
**File:** `app/(athlete)/workout/page.tsx:552-554`  
**Severity:** HIGH  
**Impact:** For users west of UTC, `new Date().toISOString().split('T')[0]` may return yesterday's date. The "today" indicator and date grid comparisons will be off by one day. The codebase already has `getLocalDateISOString` imported and used correctly elsewhere (line 67).

**Patch:**
```ts
// Lines 552-554 — replace:
const dateStr = date.toISOString().split('T')[0];
const isSelected = dateStr === selectedDate.toISOString().split('T')[0];
const isToday = dateStr === new Date().toISOString().split('T')[0];

// With:
const dateStr = getLocalDateISOString(date);
const isSelected = dateStr === getLocalDateISOString(selectedDate);
const isToday = dateStr === getLocalDateISOString(new Date());
```

**File:** `app/(athlete)/workout/page.tsx:123-128`  
**Impact:** Same timezone off-by-one as H-6.

**Patch:**
```ts
const getCurrentWeekMonday = () => {
  const d = new Date(new Date().toISOString().split('T')[0] + "T12:00:00");
  const day = d.getDay();
  const diff = d.getDate() - day + (day === 0 ? -6 : 1);
  d.setDate(diff);
  return d.toISOString().split('T')[0];
};
```

## Hold This In Your Hands

AquaFlow Pro has a solid architecture — the store with sync engine, mutation guard, and localStorage persistence is well-designed for the 10-30 user team size. The design system (deep-sea dark theme with teal accent, glass panels, water ripples) is polished and cohesive. **But three critical issues undermine the core value proposition:** plans corrupt after the second edit (C-1), any swimmer can access coach settings (C-2), and the app gets permanently stuck offline if the database quota is hit (C-3). These are all one-to-three-line fixes. Patch them and the system becomes robust enough for daily poolside use.
